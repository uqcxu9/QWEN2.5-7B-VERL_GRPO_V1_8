五、训练脚本关键配置（train_econ_grpo_qwen3_4b.sh）
=====================================================

关键环境变量：
  VLLM_SLEEP_LEVEL=0                    # 禁用 cumem allocator（CUDA driver 580.95 兼容性）
  VLLM_WORKER_MULTIPROC_METHOD=spawn
  VLLM_ATTENTION_BACKEND=XFORMERS

关键 hydra 参数（针对单卡 H100 80GB 显存优化）：
  gpu_memory_utilization=0.18           # vLLM 只用 ~14GB（模型 8GB + 最小 KV cache）
  max_model_len=864                     # 限制 KV cache 大小（prompt ~700 + response ~48）
  enforce_eager=true                    # 禁 CUDA graph
  ++enable_sleep_mode=false             # 禁 sleep mode（避免 cumem bug）
  free_cache_engine=false               # 不释放 cache（与 sleep_mode=false 配合）
  optimizer_offload=true                # Adam state 尝试 offload（单卡可能不生效）
  ++override_optimizer_config={foreach: false}  # 禁用 foreach optimizer（省临时显存）
  ++override_config.attn_implementation=flash_attention_2  # 用 flash attention

econ 专用参数：
  ++econ.enabled=true
  ++econ.num_agents=100
  ++econ.num_months=12                  # 完整年度
  ++econ.replica_group_size=4           # GRPO 4 replicas
  ++econ.rollout_agent_micro_batch=16   # 每批 16 个 agent 送 vLLM
  ++econ.rollout_prompt_length=768      # prompt 最大 token 长度
  ++trainer.val_before_train=false      # 跳过训练前验证（省时间）
  trainer.total_epochs=3                # 3 个 epoch = 60 steps

七、已知问题与解决方案
=======================

7.1 cumem_allocator CUDA Error: invalid argument
  原因：vllm 0.9+ 的 cumem allocator 和 CUDA driver 580.95 不兼容
  解决：VLLM_SLEEP_LEVEL=0 + enable_sleep_mode=false + free_cache_engine=false
  代价：vLLM 不会在训练时释放显存，需要压缩 gpu_memory_utilization

7.2 单卡 80GB OOM
  原因：4B 模型训练需要 ~62GB + vLLM ~14GB = 76GB，optimizer step 需要临时空间
  解决：gpu_memory_utilization=0.18, max_model_len=864, foreach=false
  如果还 OOM：考虑换 2 卡机器，或用 Qwen2.5-3B-Instruct

7.3 async event loop 冲突
  原因：verl 0.7.1 用 AsyncActorRolloutRefWorker，直接调 generate_sequences 会
        触发 run_until_complete 在 uvloop 里嵌套报错
  解决：EconVLLMGenerateWrapper 直接调 vLLM server handle，绕过 fsdp_workers

7.4 transformers 版本
  Qwen3 需要 transformers >= 4.51
  vllm 0.6.3 需要 transformers < 4.46
  → 必须用 vllm 0.9+

7.5 numpy pickle 兼容
  state bank pkl 必须用训练 venv 的 numpy 序列化（numpy 1.26 vs 2.x 不兼容）

7.6 opentelemetry 版本
  Ray 2.54 dashboard 需要 opentelemetry-api >= 1.29
  vllm 0.8.5 pin opentelemetry < 1.27
  → pip install "opentelemetry-sdk>=1.29" "opentelemetry-api>=1.29"
  → 或 ray start --include-dashboard=false 绕过

八、数据流总览
===============

State Bank (year N snapshot)
  → restore_env_snapshot() → env, obs, memory
  → clone 4 replicas
  → 每个 replica 跑 12 个月:
      → build_monthly_prompt_from_state() → prompt text
      → tokenize → DataProto → EconVLLMGenerateWrapper.generate_sequences()
        → vLLM server generate (Ray remote) → response tokens
      → batch_decode → parse_action_json() → env.step()
      → collect_env_stats_step() → monthly_series
      → update_memory_state()
      → 每 3 月: quarterly reflection
  → build_annual_metrics() → compute_annual_reward() → R_r + breakdown
  → _compute_group_advantages() → A_r = (R_r - μ) / (σ + ε)
  → export_grpo_samples() → tokenized samples with loss_mask
  → _stack_grpo_samples_to_dataproto() → DataProto batch

→ checkpoint_manager.sleep_replicas() （释放 vLLM，如果 sleep_mode 开的话）
→ compute_log_prob() → old_log_probs
→ compute_ref_log_prob() → ref_log_prob
→ build_replica_token_level_scores() → token_level_rewards
→ compute_advantage(adv_estimator="grpo") → advantages
→ update_actor() → GRPO policy gradient update
→ log to wandb (reward breakdown + env stats + actor metrics)

十、如果换新机器
=================
1. 如果 CUDA driver >= 590（较新），vllm 0.11 的 cumem 可能没 bug
   → 可以开 enable_sleep_mode=true, free_cache_engine=true
   → gpu_memory_utilization 可以调到 0.35+
   → 不需要 VLLM_SLEEP_LEVEL=0

2. 如果有 2 张 GPU
   → tensor_model_parallel_size=2
   → gpu_memory_utilization=0.4
   → 不需要 optimizer_offload

3. 如果有 A100 40GB
   → 4B 模型放不下，换 Qwen2.5-3B-Instruct + 旧 vllm 0.6.3 环境
